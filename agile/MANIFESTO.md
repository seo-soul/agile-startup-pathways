# Manifesto for Agile Software Development

*February 2001 — Snowbird, Utah*

---

We are uncovering better ways of developing software by doing it and helping others do it.

Through this work we have come to value:

| We value more | Over |
|---------------|------|
| **Individuals and interactions** | processes and tools |
| **Working software** | comprehensive documentation |
| **Customer collaboration** | contract negotiation |
| **Responding to change** | following a plan |

That is, while there is value in the items on the right, we value the items on the left more.

## Reading the Manifesto

A common misreading is that the right-hand items have no value. The manifesto is explicit: *"there is value in the items on the right."* The point is not to eliminate process, documentation, contracts, or planning — but to avoid letting them become ends in themselves at the expense of the left-hand values.

**Individuals and interactions** → Tools and processes serve the people using them, not the other way around. A bureaucratic process that slows down collaboration is a red flag.

**Working software** → A 200-page spec that nobody reads delivers no value. Running code that users can interact with does. Documentation should support software delivery, not substitute for it.

**Customer collaboration** → Contracts define scope; collaboration reveals what scope should have been. The goal is a shared understanding, not a legal record.

**Responding to change** → Plans are useful. But a plan executed rigidly in the face of new information is just expensive. Agility means updating your course when you learn something new.
